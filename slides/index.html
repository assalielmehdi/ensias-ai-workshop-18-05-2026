Why this matters

AI is eating real backend work

  • Most real-world AI features are not chatbots — they're automated processes with LLMs in the loop.
  • Companies are replacing tier-1 support, internal ops, and triage with these systems.
  • The interesting engineering isn't the model — it's the graph around the model: tools, routing, guardrails, evals.

Goal here is to motivate why we care. Most AI engineering jobs aren't "train a model" — they're "wire up Claude/GPT/etc. into a reliable, testable system." That's what we're going to do today.

Business problem

Support ticket overload

The pain

A small e-commerce shop. 500 tickets/day. 3 humans.

Most tickets are repetitive: where's my order, refund status, broken item.

Customers wait 24h+ for replies.

The ask

Auto-triage every ticket: classify, gather info, draft a reply, flag risky ones for a human.

Cheap, fast, safe. No bad refunds. No leaked PII.

Concrete domain matters. We're not building "an agent" in the abstract — we're solving a real problem that has constraints (no auto-approving refunds, must escalate legal threats, must not hallucinate orders).

Concept

What is an agentic workflow?

A predefined code path where each step may use an LLM or tool, but the sequence is hand-written.

  • Predictable: you can draw it on a whiteboard.
  • Easy to test: every node has a fixed input and output.
  • Limited: the LLM never decides "do I need another tool call?"

Anthropic's term, from Building Effective Agents: "workflows are systems where LLMs and tools are orchestrated through predefined code paths."

Key sentence: the sequence is hand-written. The LLM is a worker inside someone else's script.

Workflow architecture (our v1)

Workflows can branch and loop — but only along paths we wrote in code. The route node here sends refund vs. delivery tickets to different lookups. The LLM never picks what comes next; our Python does.

Important nuance: workflows are not "straight lines." They can branch, loop, retry — anything plain Python can do. What makes them workflows (and not agents) is that all of that control flow is decided by code we wrote, not by the LLM at runtime.

In our v1, the route node already does this: a refund ticket pulls different KB snippets than a delivery ticket. We just kept the diagram linear for clarity.

Concept

What is a true agent?

A system where the LLM dynamically decides what to do next:

  • Look at the current state.
  • Choose: call a tool, ask a clarifying question, or finish.
  • Observe the result.
  • Repeat — until done or a guardrail stops it.

Anthropic: "agents are systems where LLMs dynamically direct their own processes and tool usage."

The path is not hand-written. The contract is (prompt + tools + iteration cap).

The teaching message: Not every AI system that uses tools is an agent. Workflows use tools too. What makes it an agent is the loop where the model picks the next step.

Agent loop architecture (our v2)

One LLM node. One tool node. One conditional edge. Everything else is guardrails.

Two nodes, one conditional. That's it. The "intelligence" lives in the prompt and the model — not in the graph.

Build v2: the agent loop

Checkpoint git checkout step-03-agent-loop

The whole agent graph is ~30 lines in src/support_bot/agent_graph.py. The shape that matters:

def should_continue(state):
    if state["iterations"] >= MAX_ITER:    # guardrail
        return "end"
    if state["messages"][-1].tool_calls:    # LLM wants a tool
        return "tools"
    return "end"                             # LLM is done

Run it on the ticket the workflow couldn't handle:

uv run python -m support_bot.main --mode agent --ticket T-002

Walk through the conditional edge — that's the only structural difference from a workflow. The LLM's response decides whether we go to tools or to END.

Drive home: this isn't about a "smarter" model — the workflow uses the same mock model. It's that the agent's loop lets it look at what tools returned (or what's missing) and choose the next move.

Tools, observations, stopping conditions

Tools (just typed Python functions)

  • search_knowledge_base(query)
  • check_order_status(order_id)
  • check_refund_policy(...)
  • escalate_to_human(reason)

Docstrings are the LLM's manual. Spend time on them.

When does the loop stop?

  • Model emits a final message (no tool calls).
  • Model calls escalate_to_human.
  • Iteration cap hit (we set 6).

Three stopping conditions. Always include the iteration cap — agents will loop forever if you let them.

Turning the bot into a backend service

Checkpoint git checkout step-04-fastapi-entrypoint

The API is a thin, typed envelope around run_workflow / run_agent:

@app.post("/agent/tickets", response_model=TicketResponse)
def agent_tickets(req: TicketRequest) -> TicketResponse:
    return run_agent(req)

Four endpoints: GET /health, POST /workflow/tickets, POST /agent/tickets, POST /tickets (mode in body).

Then open localhost:8000/docs for an auto-generated Swagger UI.

The point: an "AI agent" in production is just a backend service. Same Pydantic models, same logging, same tests, same FastAPI patterns you'd use for anything else.

Production concerns

ConcernWhat we do
Structured outputPydantic TicketResponse on every reply
GuardrailsRefund tool returns eligibility, never approves. Safety node forces human escalation on legal/threat language.
Loop safetyHard AGENT_MAX_ITERATIONS cap; fallback escalation on overflow.
LoggingEvery LLM call and tool call → structured log line
TracingEnd-to-end view of each ticket: which nodes ran, every tool call, prompts, latency, token cost. Plug in LangSmith or OpenTelemetry — one env var and you get a UI to debug every run.
Evalsevals.py runs the same 5 tickets through both versions, prints a diff
Tests22 tests; all run in LLM_MODE=mock with zero network

Most of the engineering effort in real AI systems is here. Building the loop is the easy part; making it safe and observable is the hard part.

Tracing is what saves you when an agent does something weird in production. Logs tell you that something happened; tracing tells you why — which prompt, which tool result, how the model interpreted it.

This is the verdict slide. Read the bottom line. Many real systems are a workflow with one agentic step inside it, not a full agent.

Takeaways

  • An agentic workflow = LLM/tools inside a hand-written pipeline. Predictable, testable, limited.
  • A true agent = LLM-driven loop with tools. Flexible, but needs guardrails.
  • Not every AI system that uses tools is an agent.
  • Start with a workflow. Move to an agent only when the path depends on what tools return.
  • Production AI is backend engineering with one extra component. The boring parts (types, logging, tests, FastAPI) are still 90% of the work.
